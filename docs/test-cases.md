# Test cases

## Automated test suite

```bash
cd backend
pytest tests/ -q
```

46 tests, ~3 seconds, **zero network calls** — every LLM/embedding-touching
code path is dependency-injected with `MockLLMProvider` (always returns
`None`, forcing the deterministic fallback) and
`EmbeddedDemoPolicyRetriever` (static metadata filter, no embedding call) in
tests, per `docs/PROJECT-RULES.md` §10 ("workflow tests pass mà không gọi
live LLM"). See `tests/conftest.py` (chdir to repo root so dataset-relative
paths resolve regardless of invocation directory) and the `setUp`/
`dependency_overrides` pattern in `tests/test_api_and_actions.py` and
`tests/test_case_management_api.py`.

| File | Covers |
| --- | --- |
| `test_income_calculator_and_storage.py` | Deterministic arithmetic edge cases (rounding, missing values, currency mismatch, cross-case isolation) |

Run them against the real system (not the network-free test suite):

```bash

python3 scripts/seed_synthetic_cases.py --run --only case_08_volatile_income  # just one
```

### Why cases 4, 5, and 10 say "documented gap" instead of a stricter outcome

Cases 4 (expired contract) and 5 (contract expiring soon) were originally
written expecting `manual_review`, matching policy rule IVP-5. Running them
against the real pipeline surfaced that `app/tools/consistency_rules.py`
does not compare `contract_expiry` against the verification date at all —
so the actual, correct-per-current-code outcome is `verified` (the income
figures line up fine). Case 10 (multiple income sources) similarly assumed
a `SECONDARY_INCOME_NOTED` finding that no code path produces.

Two options existed: implement the missing checks now, or make the ground
truth honestly reflect current behavior and flag the gap. Given the scope
of this pass, the ground truth was corrected to match reality — asserting a
result the code doesn't produce would make the test suite lie about system
capability, which is exactly the kind of gap this pass exists to close, not
reintroduce. The gap itself is tracked in `docs/data-model.md`'s "Known
limitations" section as follow-up work.
